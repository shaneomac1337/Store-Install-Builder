GK Install Builder — Architecture

This page provides a technical overview of the GK Install Builder's internal architecture and the runtime execution flow of the generated GKInstall scripts.

3

System Architecture

The GK Install Builder is a Python GUI application that generates cross-platform installation packages for retail store environments. The architecture follows a delegation pattern: the main orchestrator (generator.py) coordinates specialised generator modules, each responsible for a specific aspect of the output.

Template System

Templates use @VARIABLE@ token replacement (simple string substitution, no templating engine). Templates are located in gk_install_builder/templates/ and come in platform pairs (.ps1.template / .sh.template).

GKInstall Script — Runtime Execution Flow

This diagram shows what happens when GKInstall.ps1 or GKInstall.sh runs on a retail store terminal. The script is essentially a preparation machine — it resolves identity, assembles configuration files, and then hands everything to the Launcher for the actual installation.

Phase 1: Parse & Validate

The script begins by parsing CLI arguments (-ComponentType, --storeId, --offline, etc.), validating the component type, and setting up logging with timestamped output.

Phase 2: Cleanup & Setup

Removes stale files from previous runs, creates the logging directory, and prepares the working environment.

Phase 3: Fresh Install vs. Update

The script checks for an existing station.properties file in the installation directory:

PathWhat Happens
Fresh Install
(no station.properties)		 Loads environments.json, selects environment, applies credentials and tenant ID from environment config.
Update Mode
(station.properties found)		 Reads Store ID, Workstation ID, tenant ID, and Config Service URL from the existing file. Skips environment selection. CLI parameters (Priority 0) can still override.

Phase 4: Version Resolution

Determines the component version to install. Sources checked in order:

  1. CLI --VersionOverride parameter
  2. API fetch (Config-Service or Function Pack)
  3. Hardcoded defaults from the generated script

Phase 5: Identity Detection

Resolves Store ID and Workstation ID using the five-tier priority system (see Detection Priority table above). For fresh installs, this may involve hostname regex matching or reading .station files.

Phase 6: Onboarding & Store Initialisation

This phase runs only for fresh installs. Updates skip onboarding entirely.
  • Onboarding: Calls onboarding.ps1 / .sh to generate OAuth2 token and obtain a CIMS onboarding token
  • Store Initialisation: Calls store-initialization.ps1 / .sh to register the workstation in Store Manager and assign it to SDC

Phase 7: Assemble Installation Token

Generates the installation token file containing Config Service URL, tenant ID, onboarding token, DSG URL, and other configuration. This is the primary input for the Launcher.

Phase 8: Launcher Setup

  • Processes SSL certificate and writes launcher.properties
  • Resolves the Launcher executable URL and downloads Launcher.exe / Launcher.run (or uses offline copy)
  • Generates installer_overrides.xml if overrides are configured

Phase 9: Execute Launcher

text

Phase 10: Monitor & Complete

The script monitors installer.log for 3-hour timeout, checks for the revert-installation-logic marker (to catch silent rollbacks), and then:

OutcomeAction
Success Copies logs to results/, cleans up tokens and override files, exits with code 0
Failure Copies logs to errors/, preserves installer output for debugging, exits with code 1
