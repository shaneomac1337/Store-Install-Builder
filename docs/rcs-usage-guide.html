GK Install Builder — Resource Cache Service (RCS) Integration

Overview

What is RCS?

The Resource Cache Service (RCS) acts as a local cache layer inside a store that serves installation packages to other components, reducing bandwidth to the cloud DSG. Instead of every POS/ONEX etc. node downloading from the internet, they download from the local RCS.

The RCS implementation in GK Install Builder covers three areas:

1 RCS as an installable component — installing the RCS Service itself on a machine
2 RCS URL configuration — registering the RCS URL in Config-Service so other nodes discover it
3 RCS URL in installation token (--rcsUrl) — telling the Launcher to download packages from RCS instead of from cloud DSG

RCS URL Mode — Hostname vs IP Address

#4c9aff #deebff solid

Config key: rcs_url_mode | Default: Grey hostname

Controls whether the registered RCS URL uses the machine hostname or its IP address. Two radio buttons in the RCS-SERVICE tab let you choose the mode:

Mode Host Component HTTPS Compatible Resulting URL
Green hostname (default)
$env:COMPUTERNAME (Windows) / hostname (Linux) Yes http://<hostname>:8180/rcs
Yellow ip
IP resolved at runtime via the default gateway adapter No — HTTPS is automatically disabled http://<ip-address>:8180/rcs

In IP mode, the store-initialization script resolves the machine's IP address at runtime by inspecting the network adapter associated with the default gateway. HTTPS is automatically disabled in this mode because SSL certificates cannot validate raw IP addresses.

Hostname mode is the recommended default. Only switch to IP mode for edge-case environments where stable DNS is not available, or for example having problems with hostname with underscores like project ACH (Action). Using IP mode in environments with dynamic IPs will break the registered URL after a reboot or DHCP lease renewal.

3. Store-Initialization — RCS URL Registration

When the generated store-initialization script runs for an RCS-SERVICE component, it registers the RCS URL in Config-Service so that other nodes in the store can discover it.

Flow

1 Checks that the component type is RCS-SERVICE and rcs_skip_url_config is not "true"
2 Reads the update_config.json template from helper/init/rcs/
3 Gets the machine hostname ($env:COMPUTERNAME on Windows, hostname on Linux)
4 Constructs the RCS URL: <protocol>://<hostname>:<port>/rcs
5 Replaces all placeholders in the config template
6 POSTs the config to Config-Service API

API Endpoint

text /api/config/services/rest/config-management/v1/parameter-contents/plain]]>

Request Body (update_config.json)

json " }, "systemDescriptor": { "systemName": "GKR-Resource-Cache-Service", "systemVersionList": [ { "name": "" } ] }, "user": "", "parameterValueChangeList": [ { "name": "system.properties", "url": "rcs.url", "value": "http://:8180/rcs" } ] }]]>

This sets the rcs.url property inside system.properties for the RCS node in Config-Service. Other components use this property to locate the RCS instance in the store.

Template Placeholders

Placeholder Replaced With Set By
@RCS_PROTOCOL@ http or https helper_file_generator.py (from rcs_use_https config)
@RCS_PORT@ 8180 or 8543 (or custom) helper_file_generator.py (from launcher settings)
@RCS_SKIP_URL_CONFIG@ true or false helper_file_generator.py (from rcs_skip_url_config config)
@RCS_URL@ Full URL at runtime Replaced by the store-init script itself at install time

4. The --rcsUrl CLI Parameter

Purpose

The --rcsUrl parameter on the generated GKInstall script tells the Launcher to download installation packages from a local RCS instead of from the cloud DSG. It (a) appends rcs.url=<value> to the installationtoken.txt file, and (b) injects the resolved URL into each non-RCS component's onboarding token request via afterOnboardingProperties (see below).

Three Modes of Operation

Mode Usage Behavior
Grey Not specified
.\GKInstall.ps1 Normal behavior. The Launcher downloads from DSG using the dsg.url in the installation token.
Green Direct URL
.\GKInstall.ps1 -rcsUrl http://10.63.2.215:8180/rcs Appends rcs.url=http://10.63.2.215:8180/rcs to the installation token. The Launcher uses RCS for package downloads.
Blue Autodetect
.\GKInstall.ps1 -rcsUrl autodetect Queries Config-Service to find the RCS URL for this store, then appends it to the installation token. Falls back to user prompt if detection fails.

Installation Token

The base installation token always contains:

text installationtoken.txt (base) /api/config cims.url=https:///api/iam/cim/rest station.tenantId= station.storeId= station.workstationId= station.applicationVersion= station.systemType= onboarding.token= dsg.url=https:///api/digital-content/content/cep/SoftwarePackage]]>

When --rcsUrl is provided, the following line is appended after the token file is written:

text Appended line

The complete token is then base64-encoded as installationtoken.base64 for the Launcher.

afterOnboardingProperties Injection

In addition to the installationtoken.txt append, when --rcsUrl is set (direct URL or autodetect) the resolved RCS URL is injected into the onboarding token request body as an afterOnboardingProperties entry, so the component receives it as part of the signed onboarding token.

json Onboarding request body (non-RCS component, rcsUrl set)
Aspect Behavior
Scope All non-RCS components (POS, WDM, Flow, LPA, StoreHub, MQTT-Broker). The RCS-SERVICE onboarding body is left unchanged — it does not need its own URL.
Resolution order GKInstall pre-acquires the OAuth token and resolves the RCS URL (literal or autodetect) before the onboarding call, so the value is available for injection.
In-memory only The on-disk helper/onboarding/*.onboarding.json files are not modified — injection happens on the request body only.
Belt-and-suspenders The existing installationtoken.txt rcs.url= append is preserved unchanged. Both paths carry the value.

Update-Mode RCS Detection

During an update (existing station.properties present), if that file already contains an rcs.url entry, the progress message now reads "Downloading installation files from RCS (<url>)" instead of the misleading "... from <base_url> DSG". This is cosmetic only — no token changes, no side effects; the installer already used RCS automatically in this case. Precedence: CLI --rcsUrl > station.properties rcs.url > DSG fallback.

5. Autodetect — How It Works

#6554c0 #eae6ff solid

When --rcsUrl autodetect is used, the script performs two API calls to Config-Service to find the rcs.url property that was previously registered by the store-initialization script (see Section 3).

Step 1 — Get Store Child Nodes

text API Call 1 /api/config/services/rest/infrastructure/v1/structure/child-nodes/search]]> json Request Body ", "retailStoreId": "" } }]]>

The response contains a childNodeList with all nodes registered under this store. The script finds the node whose systemName contains Resource-Cache-Service.

Step 2 — Fetch system.properties

text API Call 2 /api/config/services/rest/config-management/v1/parameter-contents/plain/search]]> json Request Body " }, "systemDescriptor": { "systemName": "", "systemVersionList": [ { "name": "" } ] }, "parameterList": [ { "name": "system.properties" } ] }]]>

Step 3 — Extract the URL

The system.properties content is returned as a base64-encoded string. The script:

a Decodes the base64 content
b Parses the rcs.url=... line using regex
c Handles Java-style property escaping (\: → :, \= → =)

Authentication

Both API calls use a Bearer token read from helper/tokens/access_token.txt. When --rcsUrl is set, GKInstall pre-acquires this OAuth token and runs autodetect before the onboarding step (so the resolved URL can be injected into the onboarding token via afterOnboardingProperties). The subsequent onboarding phase re-acquires its own token normally; the pre-acquired file is harmlessly overwritten.

Failure Handling

If autodetection fails at any step, the script prompts the user:

text Possible Failure Reasons
  • No access token available for RCS URL autodetect — token file missing
  • Failed to fetch store child nodes — Config-Service API unreachable
  • No Resource-Cache-Service node found for store <id> — RCS not registered for this store
  • No system.properties content returned for RCS node — properties not configured
  • system.properties found but no rcs.url entry — property exists but rcs.url line missing

7. Typical Deployment Scenarios

Scenario A — First RCS Installation in a Store

#00875a #e3fcef solid
  1. Generate installation files with the GK Install Builder
  2. Run GKInstall.ps1 -ComponentType RCS on the RCS server
  3. The Launcher downloads the RCS Service from DSG (cloud)
  4. The store-initialization script registers rcs.url=http://<hostname>:8180/rcs in Config-Service

Scenario B — Installing POS via RCS (Direct URL)

#0747a6 #deebff solid
  1. RCS is already installed and its URL is known
  2. Run GKInstall.ps1 -ComponentType POS -rcsUrl http://10.63.2.215:8180/rcs
  3. The installation token gets rcs.url=http://10.63.2.215:8180/rcs
  4. The Launcher downloads POS packages from RCS instead of DSG

Scenario C — Installing POS via RCS (Autodetect)

#6554c0 #eae6ff solid
  1. RCS is already installed and registered in Config-Service
  2. Run GKInstall.ps1 -ComponentType POS -rcsUrl autodetect
  3. The script queries Config-Service, finds the RCS node, extracts rcs.url
  4. The installation token gets the autodetected rcs.url
  5. The Launcher downloads POS packages from RCS

Scenario D — RCS without URL Registration

#ff8b00 #fffae6 solid
  1. Enable "Don't set RCS URL" in Launcher Settings
  2. Generate and run the installation
  3. RCS is installed but rcs.url is not registered in Config-Service
  4. Useful when the URL is managed externally or via a different mechanism or just manually in SM
