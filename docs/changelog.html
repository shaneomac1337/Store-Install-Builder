New Features

New Component: Store MQTT Broker

A new installable component, Store MQTT Broker, is now available end-to-end (UI, version management, onboarding, launcher, API integration, offline packaging) on Windows and Linux.

  • System Type: GKR-Store-MQTT-Broker

RCS URL Injection via afterOnboardingProperties

When --rcsUrl is supplied (a literal URL or autodetect), the resolved RCS URL is now embedded directly into each non-RCS component's onboarding token request as an afterOnboardingProperties entry, not only appended to installationtoken.txt.

  • Scope: All non-RCS components (POS, WDM, Flow, LPA, StoreHub, MQTT-Broker). RCS-SERVICE onboarding body left unchanged — it does not need its own URL
  • Resolution Order: GKInstall now pre-acquires the OAuth token and resolves the RCS URL (Config-Service system.properties autodetect) before the onboarding call so the value is available for injection
  • In-Memory Only: The helper/onboarding/*.onboarding.json files on disk are not modified — injection happens to the request body only
  • Belt-and-Suspenders: The existing installationtoken.txt rcs.url= append behavior is preserved unchanged

Update-Mode RCS Download-Source Detection

During an update, if the existing station.properties contains an rcs.url entry, the progress message now reads "Downloading installation files from RCS (<url>)" instead of the misleading "... from <base_url> DSG".

  • Cosmetic Only: No change to $rcsUrl auto-set, no installationtoken.txt append, no onboarding side effects — the installer already used RCS automatically in this case; only the displayed source was wrong
  • Precedence: CLI --rcsUrl > station.properties rcs.url > DSG fallback
  • Both Platforms: Implemented in GKInstall.ps1 and GKInstall.sh with Java-property colon unescaping

Bug Fixes

Bash sed Fallback Broke on RCS URLs Containing Slashes

The no-jq sed fallback used to inject afterOnboardingProperties used / as the substitution delimiter; every real RCS URL contains / (e.g. https://host:7443/rcs), which broke the command.

  • Fix: Switched the sed delimiter to | and anchored the substitution to the last line only (avoids matching nested closing braces); added an empty-result guard plus a stderr warning
  • Linux Autodetect: Follow-up fix corrected the no-jq Bash fallback path for the RCS URL autodetect Config-Service parse on Linux hosts without jq

API Test Summary Showed "8/7 Components Found"

The Config-Service / Function-Pack API test result dialog hardcoded the component denominator to 7. After MQTT-Broker was added as the 8th component, the summary read an impossible 8/7 components found.

  • Fix: Replaced the four hardcoded /7 occurrences with the dynamic component count so the denominator can never drift again

Offline Package: Two Tomcat/Java Versions Pre-Selected

The Tomcat/Java file picker pre-selected two entries when installer.properties specified a preferred version: the preferred file and the latest available file. The latest-file auto-pick fired silently for any file not already preferred.

  • Fix: When installer.properties supplies an explicit preference, the auto-latest fallbacks (Java and non-Java) are skipped — the preferred file wins, no silent duplicates
  • Was Hidden for Java: The same latent bug existed for Java; it was invisible only because the preferred Java version happened to equal the latest available

Testing

  • New test_rcs_afteronboarding.py: Covers afterOnboardingProperties injection (PS1 + SH, RCS-SERVICE exclusion), the sed-delimiter regression, GKInstall pre-onboarding rcsUrl resolution ordering, update-mode station.properties RCS detection, and end-to-end flow consistency across the generated scripts
  • Suite Status: 422 passing (1 pre-existing unrelated pytest-mock setup error, untouched by this work)


Build 5.27-b10 - May 2026

PRIA-1740 RCS-SERVICE version-parse fix (PowerShell single-item list returned 'v') + RCS-SERVICE FP/FPD dispatch coverage across PS1, Bash, and GUI test

Bug Fixes

PRIA-1740 — RCS Default Version Parse Returned Single Character 'v'

Generated GKInstall.ps1 resolved the RCS-SERVICE version to the literal character v instead of the full version string (e.g. v2.1.10) when Config-Service returned a single-element versionNameList. Downstream API calls and DSG download URLs then 404'd with "System type version 'v' for system 'GKR-Resource-Cache-Service' does not exist."

  • Root Cause: PowerShell pipeline unwraps a single-element Sort-Object result to a scalar string; the trailing [0] index then returned [char]'v' instead of the first element
  • Fix: Replaced [0] indexing with Select-Object -First 1 in GKInstall.ps1.template
  • Scope: Both Legacy (5.25) and New (5.27+) API modes were affected — same parsing code. Bash (grep | tr | sort -V | head -1) was not affected
  • Impacted Component: Most exposed on RCS-SERVICE because it typically publishes only a single version, but the fix covers any single-version component on Windows

RCS-SERVICE FP/FPD Dispatch Missing in 5 of 6 Parsers

RCS-SERVICE was listed as a target component for the Employee Hub Function Pack API but had no property-name dispatch in 5 of 6 parsing sites. The code silently fell back to the hardcoded @RCS_VERSION@ template value (often stale) instead of the live API value.

  • Sites Fixed: PS1 FP, PS1 FPD, Bash FP, Bash FPD, and Python FPD (GUI API test). Python FP was already correct
  • Property Names: RCS_Version preferred, RCS_Update_Version fallback — matching the convention used by ONEX, WDM, FlowService, LPA
  • Origin: Bug introduced when RCS-SERVICE was first added — the component was wired into the $allComponents array and mapping switches but the FP/FPD property dispatch was never added
  • Symptom: RCS installs configured to use FP/FPD as the version source used whatever stale value was in the GUI override rather than the live cloud value, with no clear error

Testing

  • PRIA-1740 Regression Test: Asserts that GKInstall.ps1.template uses Select-Object -First 1 rather than [0] indexing for the version-name parse — guards against reintroduction of the single-element unwrap bug
  • FP/FPD Dispatch Coverage Matrix Test: New test_fp_fpd_coverage.py walks each of the 6 dispatch sites and asserts every component has both *_Version and *_Update_Version property names — catches the same class of bug for any future component addition


Build 5.27-b09 - May 2026

SkipStoreInit CLI flag, runtime tenant_id propagation fix for legacy API multi-env

New Features

SkipStoreInit CLI Flag

New -SkipStoreInit (PowerShell) / --SkipStoreInit (Bash) switch added to the generated GKInstall scripts for bypassing the store-initialization step after onboarding.

  • Use Case: Edge-case deployments where the backend already has the workstation configured and a re-init would be redundant or harmful
  • Onboarding Still Runs: Only the store-initialization.{ps1,sh} invocation is skipped — onboarding completes normally
  • Default Off: Existing behavior preserved for installations that do not pass the flag
  • ⚠ Use With Caution: Only use this flag when you know exactly what you are doing. Skipping store-initialization without proper backend preparation can cause the workstation to not be registered correctly — most commonly resulting in the WSID not being installed properly on the component.

Bug Fixes

  • Legacy API — Runtime Tenant ID Propagation (PR #5): Fixed a 404 on Get Business Unit in legacy API mode (5.25) when the runtime tenant differed from the build-time tenant (multi-env installs or station.properties override). The legacy /swee-sdc/tenants/{id}/... URLs in store-initialization had the tenant baked at generation time as a Python f-string, while GKInstall correctly switched the runtime tenant for OAuth and onboarding. The runtime tenant_id is now forwarded from GKInstall to store-initialization, mirroring the pattern already used for onboarding.ps1. New API mode is unaffected (no tenant in URL path); single-env legacy installs are also unaffected because the default value is the build-time tenant.

New Features

CLI Override Parameters (Generated Scripts)

Five new CLI parameters have been added to the generated GKInstall and store-initialization scripts. All parameters are optional and take highest priority (Priority 0) over any other resolved value, making them well-suited for wrapper scripts and automated pipelines.

  • --VersionOverride — Overrides the component version resolved by the script at runtime. Accepts both prefixed (v5.27.1) and plain (5.27.1) version strings. Supported on both PowerShell and Bash.
  • --SystemNameOverride — Overrides the system name sent to the Store Manager API during store-initialization. Useful for deployments where the registered system type must differ from the component type embedded in the script. Allows you to use any inherited system type name like GKR-OPOS-ONEX-CLOUD-SCO and not just GKR-OPOS-ONEX-CLOUD.
  • --WorkstationNameOverride — Overrides the workstation name sent to the Store Manager API. Accepts any string value (e.g., A319_OneXPOS1). Propagated automatically from GKInstall to store-initialization when provided.
  • --StructureUniqueNameOverride — Injects a structureUniqueName field into the newNode object when creating workstation structures via POST /structure/nodes. Only active when explicitly passed; default behavior is unchanged.
  • -y / --yes — Auto-confirms the workstation creation prompt in store-initialization. Essential for fully non-interactive deployments where no operator is present. Automatically forwarded from GKInstall.

SSL Password Control

The SSL certificate password is now configurable and overridable at runtime without regenerating the package:

Detection Settings Dialog Readability (Windows)

  • OAuth2 Bearer Token Authentication: Implemented secure OAuth2 authentication replacing Basic Auth, with automatic token generation and refresh on 401 errors
  • Modern File Browser: Complete UI redesign with CustomTkinter integration:
    • Color-coded file types (folders, executables, archives)
    • Right-click context menu with "Copy Download URL" and "Download File" options
    • Performance optimization using lightweight Listbox instead of ScrollableFrame
    • Background threading for downloads with progress tracking
    • Downloads saved to downloaded_packages/ folder

Multi-Environment and Multi-Tenancy Support

  • Environment Manager UI: New interface for managing unlimited environments (TEST, PROD, QA, etc.) with per-environment configuration:
    • Separate Base URLs, Tenant IDs, and credentials per environment
    • Clone functionality for easy multi-tenant setup with auto-incrementing tenant IDs
    • Base64-encoded credentials in generated environments.json
  • Smart Environment Detection: Automatic environment selection with priority-based logic:
    1. CLI parameter (-Env or --env)
    2. Hostname detection using regex patterns (e.g., P-001-101)
    3. Station file detection (reading Environment= from .station files)
    4. Interactive selection with option to use default or configured environments
  • 3-Group Hostname Pattern: Enhanced regex detection ^(.+)-(.+)-(.+)$ to extract Environment-Store-Workstation from hostnames
  • Toggle Controls: Enable/disable environment detection from station files independently of store/workstation detection
